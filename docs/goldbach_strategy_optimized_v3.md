## 〇、先把问题"定性"对：这是加法问题，不是乘法问题

原始思路里夹了"密码学家""分解 $p,q$"这类措辞，容易让人把两件**难度天差地别**的事混为一谈：

| 问题 | 形式 | 难度 |
|---|---|---|
| 哥德巴赫分拆 | 找 $p+q=N$（**加法**） | **容易**：期望多项式时间，实测千位数毫秒级 |
| 整数分解（RSA） | 找 $p\cdot q=N$（**乘法**） | **困难**：至今无多项式算法，是密码学安全的基石 |

它们只是长得像。**找 $p+q=N$ 根本不"硬"**，所以我们追求的不是"破解难题"，而是"把一个本就容易的搜索做到极致优雅"。摆正这一点，后面的策略才不会跑偏。

---

## 一、最重要的一条规律：最小素数解极小（全篇地基）

把"从小到大试 $p=3,5,7,\dots$"称为"太笨"是**反的**。实测表明：

- 随机 100 位偶数，最小可用 $p$ 的**中位数约 421**，40 次实验最大也才 2297；
- 随机 300 位偶数，中位数约 1999；
- 随机 **1000 位**偶数，最小 $p$ 也只落在 6000～32000 量级（即前几千个素数之内）。

也就是说，**几乎任何大偶数，从小往大扫，扫过前几千个素数就一定撞上解。** 这符合 $p_{\min} \approx O((\ln N)^2)$ 的增长规律。

**结论：** 升序搜索本身就是接近最优的搜索*顺序*。要优化的**不是换搜索方向，而是让"扫过的前几千次判断"每一次都尽量便宜**。"精准制导第一发命中"是不存在的幻觉——你躲不开"扫几千个候选"，但可以让其中 99% 的扫描只花几个时钟周期。

> **这条规律还有一个被低估的推论（本版重点）：** 因为解出现得极早，**搜索几乎总在第一个很小的区间里就提前终止**。这直接决定了"该优化什么、不该优化什么"——见第三节对工程优化方案的甄别。

## 二、三个真正的优化杠杆

### 杠杆 1：同余过滤（轮筛）—— 它和哈代-李特尔伍德是同一件事

要 $N-p$ 是素数，至少要它不被小素数整除。对每个小素数 $r$：

$$N-p \not\equiv 0 \pmod r \;\Longleftrightarrow\; p \not\equiv N \pmod r$$

再加上 $p$ 自己也不能被 $r$ 整除（$p\not\equiv 0$）。于是对每个 $r$，$p$ 被**禁掉两个剩余类**：$0$ 和 $N\bmod r$。用 CRT 把前几个素数的禁区叠起来，就得到一张"安全步长表"，照着取 $p$，得到的 $N-p$ 必不被这些小素数整除。

实测（筛掉 $3,5,7,11$，模 $1155$）：

| $N$ | 存活 $p$ 剩余类 | 被剪掉 |
|---|---|---|
| $2^{10}$（贫瘠） | 11.7% | **88.3%** |
| $2\,000\,000$（一般） | 15.6% | 84.4% |
| $210=2\cdot3\cdot5\cdot7$（丰富） | 37.4% | 62.6% |

**串起杠杆 1 与杠杆 2 的钥匙：** 当 $r \mid N$ 时，两个禁区 $0$ 与 $N\equiv0$ **重合**，只禁掉一类而非两类 $\Rightarrow$ 存活的 $p$ 更多 $\Rightarrow$ 解更密。这恰好就是哈代-李特尔伍德奇异级数里的修正因子

$$r(N) \sim 2C_2 \!\!\prod_{\substack{r\mid N\\ r>2}}\!\frac{r-1}{r-2}\cdot\frac{N}{(\ln N)^2}.$$

所以"轮筛存活率"和"分拆解的密度"本质是同一个数学结构的两面。**同余过滤就是哥德巴赫密度公式的算法化身。**

### 杠杆 2：素因子越多解越密（"哥德巴赫彗星"）——实测确认

| $N$ | 奇素因子 | 分拆组数 |
|---|---|---|
| $65536=2^{16}$ | 无 | 435 |
| $66000$ | $3,5,11$ | **1284** |
| $69300$ | $3,5,7,11$ | **1578** |

奇素因子越多，解越密。**工程含义：** 拿到 $N$ 先试除提取小素因子。若 $N$ 富含小素因子，闭眼升序搜，第一发命中概率极高；若 $N$ 形如 $2^k$ 这种"贫瘠数"，解相对稀疏，要有耐心扫更长一截（但仍只是从几百扫到几万，依旧便宜）。

### 杠杆 3：分级素性测试 —— 把贵的测试留到最后

千位数做一次素性测试要花真金白银的 CPU 周期。所以排成漏斗：

1. **双筛批量试除**：用第三节的双筛把"$N-p$ 被小素数整除"的 $p$ 整条划掉，摊销成本接近 $O(1)$/候选。这一步**就是**对 $N-p$ 的小素数试除，无需再单列。
2. **重型概率素性测试**：只有双筛幸存者才进来。千位数用 **BPSW**（无已知反例）或 ~40 轮 Miller–Rabin；要确定性可上 AKS，但工程上没必要。

> **一个易犯的冗余（修正附件建议）：** 不要在 BPSW 之前再加一道"base-2 强伪素数"预过滤。**BPSW 的定义本身就是"先一轮 base-2 强 Fermat，再一轮强 Lucas"**，前置 base-2 等于把同一步做两遍，零收益、略慢。真正的廉价前置过滤是双筛（第 1 步），它已经做完了。

## 三、工程最优解：自适应分段"双筛"（把三个杠杆熔成一炉）

逐个 $p$ 套漏斗仍是"零售"。**批发**做法更优：在区间上同时筛 $p$ 和 $N-p$。关键观察：随 $p$ 在区间上推进，"$r \mid (N-p)$"等价于"$p \equiv N \pmod r$"——这是一条等差数列，可以像埃氏筛一样整条划掉。

```text
function goldbach(N):                       # N 为偶数, N >= 6
    if is_prime_BPSW(N-2): return (2, N-2)   # 边界先试 p=2

    R = 100000                               # 双筛的小素数上限(见下方"如何选 R")
    small_primes = primes_up_to(R)           # 一次性预筛, 多个 N 可复用

    # —— 唯一一次大数运算: 预计算 N mod r, 之后全部降为 64 位整数 ——
    rem = { r : int(N mod r) for r in small_primes }     # 采纳: 大数→本地整数降维

    B    = 4096                              # 初始区间宽度(小! 因 p_min 通常极小)
    base = 3
    while True:
        # 1) 标记 [base, base+B) 内哪些 p 是素数(标准分段埃氏筛)
        is_p_prime = segmented_prime_sieve(base, B, small_primes)

        # 2) 第二条筛: 标记 "N-p 被某个小素数整除" 的 p
        Nmp_comp = bitarray(B, false)
        for r in small_primes:               # r 一直筛到 R(激进, 见下)
            first = ((rem[r] - base) mod r + r) mod r     # 纯 64 位运算, 无大数
            for i in range(first, B, r):
                Nmp_comp[i] = true

        # 3) 升序扫描; 仅对"两条筛都幸存"的极少数 p 触发重型测试
        for i in range(0, B):
            if is_p_prime[i] and not Nmp_comp[i]:
                p = base + i
                if is_prime_BPSW(N - p):     # 唯一真正昂贵的一步
                    return (p, N - p)

        base += B
        B    *= 2                            # 极罕见: 区间不够才扩张并继续
```

要点：

- **第二条筛 `Nmp_comp`** 把"$N-p$ 显然是合数"的 $p$ 整条划掉，等价于杠杆 1+2 的轮筛，摊销成本接近 $O(1)$/候选。
- **真正的重活 `is_prime_BPSW` 只对双筛幸存者触发**——由第一节，这种 $p$ 通常几十个以内就出解。**这几十次大数 BPSW 就是全程的瓶颈。**
- **预计算 `rem[r]`**：筛阶段唯一的大数操作是 $N \bmod r$。开局一次性算好、之后 `first` 全是 64 位运算，避免在内层循环里反复做大数取模。

**如何选 $R$（修正附件的 $r_{\max}=B/20$）：** 在本场景里每省下一次 BPSW 都极其值钱（毫秒级），而多筛一个素数只多几次标记（纳秒级）。由 Mertens 定理，筛到 $R$ 后幸存密度 $\sim e^{-\gamma}/\ln R$，把 $R$ 从 $10^4$ 提到 $10^5$ 还能再砍约 15% 的 BPSW 调用，几乎免费。**所以应当激进地筛，$R$ 取 $10^4\sim10^6$**（可随 $\ln N$ 略增），而不是压到 $B/20$。注意：筛界 $R$ 与区间宽度 $B$ 是**两个独立的量**，不应绑定。

## 三·补、关于"极致工程优化"的甄别：先认清你在哪个 regime

有一类常见建议——**位图 Bitset、对齐 L1 Cache、AVX/SIMD 批量位与、轮筛**——技术上都正确，但**是否有用完全取决于场景**：

| | Regime A：验证**所有** $N \le 4\times10^{18}$ | Regime B：对**单个**千位 $N$ 找一组分拆（本文） |
|---|---|---|
| $N$ 的规模 | 64 位以内的小数 | 数千比特的大数 |
| 素性判定成本 | 近乎免费 | **每次 BPSW ~毫秒级，全程主导** |
| 是否提前终止 | 否，要扫满整个范围 | **是，$p_{\min}$ 极小，第一小块就出解** |
| 瓶颈 | **筛吞吐量** | **几十次大数 BPSW** |
| 位图 / L1 对齐 / SIMD / 轮筛 | **对，且是关键**（$4\times10^{18}$ 记录正是这么跑的） | **基本无用**：筛只占总时间 $<0.1\%$ |

**一笔账（千位数 $N$）：** 预筛小素数 + 标记一个几千宽的小区间，总计 $<1\,\text{ms}$；而几十次 BPSW（每次 $\sim3\times10^7$ 量级字操作）合计几百 ms。**筛占总运行时间不到 0.1%。** 把这 0.1% 用 SIMD 提速 5 倍，对总时间无可感知影响。

因此对 Regime B：

- **不要**把区间撑到 26 万去"喂满 L1"。那等于先把 26 万位的双筛全建出来、再在第几百位找到答案——纯过度计算。**区间应小且自适应**（开局几千，不够再翻倍）。
- **不要**用 SIMD 一次算满整块的幸存者位图。提前终止要的是"尽快拿到前几个幸存者"（低延迟），不是"把整块幸存者全算出来"（高吞吐）。
- Bitset / 轮筛在此只是无害的小常数优化，可做可不做；小区间用 `bytearray` 更简单、同样全程驻留缓存。
- **唯一值得认真采纳的底层技巧**是上面已并入的"预计算 $N \bmod r$、筛内算术全 64 位化"。

> 一句话：**这些 SIMD/缓存技巧是 Regime A 的利器；移植到 Regime B 是在优化一个本就免费的环节。** 想发力，请发在减少与加速 BPSW 调用上（更激进的双筛筛界、FFT 大数乘法、好的 BPSW 实现）——具体怎么发，见下一节。

## 三·末、向"几十次 BPSW"开炮：下一级优化的三个候选

第三节已把瓶颈钉死在"**几十次（量级上可达上百次）千位大数 BPSW**"上；第三·补节澄清了 SIMD/缓存那一类是 Regime A 的利器、在此无用。那么 Regime B 真正的下一级火力该往哪打？方向只有一个：**减少 BPSW 的次数、或压低每一次 BPSW 的成本**。下面逐条甄别三个常见的"终极优化"建议。

### 候选 1：蒙哥马利模约（Montgomery / REDC）—— ✅ 正确，应采纳

BPSW 的两步（base-2 强 Miller–Rabin 的模幂、强 Lucas 序列的递推）内层全是"大数模乘 $A\cdot B \bmod m$"。传统实现里取模靠**高精度试除**，而除法在大数上比乘法贵得多。

关键前提**恰好成立**：单次 BPSW 测试中，模数 $m=N-p$ 是**固定不变**的，且 $N-p$ 为**奇数**（$N$ 偶、$p$ 为奇素数）——这正是 Montgomery 模约的适用条件。把操作数一次性变换进"蒙哥马利域"，此后每次模约只剩**移位、乘法与按位与**，把除法彻底消掉。进出蒙哥马利域的变换开销，在一次模幂的 $\sim\log_2 m\approx$ 数千次平方里被充分摊销。

- **收益**：单次 BPSW 的常数因子下降**数倍**（视基线而定，约 $2\sim4\times$；对照最朴素的"除法取模"基线时更明显）。
- **诚实的边界**：成熟大数库（如 GMP / gmpy2 的 `mpz_powm`）**内部已在使用** Montgomery/REDC，此时这部分收益已被吃掉；真正的"质变"发生在**手写 C++/Rust**、或所用路径仍以试除取模时。此外，千位数（$\approx 3300$ 比特）的大数乘法尚在 Toom–Cook 区间（FFT 乘法要到数万比特才占优），所以**乘法本身**也是大头，Montgomery 优化的是其中"模约"那一环，而非全部。
- 与既有结论一致：这就是第三·补节末尾所说"好的 BPSW 实现"的**具体落地**。

### 候选 2：多核并行的"彩票扫荡" —— ✅ 正确，应采纳

双筛会**瞬间**吐出一小批幸存候选 $p_1,p_2,\dots,p_k$（由第一节，到撞上素数前通常只需检验几十个）。串行地一个个 BPSW，若前若干个的 $N-p$ 都是合数，就只能干等。

因为**哥德巴赫只要任意一组解、并不要求 $p$ 最小**，所以可以放心地**推测式并行**：开 4–8 个线程，把队首的若干幸存者同时丢进 BPSW；**谁先验出素数，就用一个全局取消令牌（cancellation token）掐断其余线程并立即返回**。把第三节伪码的步骤 3 改成"先收集一小批幸存者、再并行派发"即可：

```text
        # 3') 收集一小批双筛幸存者, 推测式并行验素
        survivors = [ base+i for i in range(0,B)
                      if is_p_prime[i] and not Nmp_comp[i] ]   # 升序, 通常仅几十个

        result = parallel_race(survivors, fn = p -> {          # std::async / Rayon
            if is_prime_BPSW(N - p): return (p, N - p)         # 谁先命中谁赢
        })                                                     # 命中即取消其余线程
        if result is not None: return result
```

- **效果**：BPSW 阶段的**墙钟时间**约降到 $1/\text{核数}$（更准确地说，受 $\min(\text{核数},\ \text{所需调用次数})$ 限制）。由于 BPSW 阶段几乎就是全程，墙钟总时间近线性缩短。
- **诚实的边界**：这是**降墙钟延迟**而非降总 CPU 功——推测式并行会多算几次注定要被取消的 BPSW（轻微"超射"）。但 Regime B 是"对单个 $N$ 求一组解"的**低延迟**任务，延迟正是我们要的；若 $N$ 富含小素因子、一两发就命中，多开的线程白跑但也不拖慢延迟。

### 候选 3：非对称 BPSW（先全员跑 base-2、再统一上 Lucas）—— ❌ 基本冗余，不另立

这条建议察觉到一个**真实**的不对称：base-2 强 Miller–Rabin 只需一次模幂（快），而强 Lucas 涉及雅可比符号与序列递推（约为前者的 $2\sim3$ 倍）。由此提议：先让**所有**幸存者跑完第一步 base-2、筛掉合数，再只对幸存者跑昂贵的 Lucas，以免"昂贵的 Lucas 被意外唤醒"。

问题在于：**任何正确实现的 BPSW 本就是短路的**——它先做 base-2 强 MR，**只有通过者才进入 Lucas**。而对随机大合数，base-2 强 MR **几乎必然当场判合**（base-2 强伪素数极其稀疏，最小者是 $2047$），于是在你那一批幸存者里，所有合数都被 base-2 当场拦下，**Lucas 实际上只会落在那个真正的素数身上**。也就是说，"重排成先全员 base-2、再统一 Lucas"省下的 Lucas 调用约等于**零**——标准短路 BPSW 已经把这件事做完了。

几点订正与提醒：

- 建议中举的 $341$ 并不贴切：$341=11\times31$ 只是 base-2 **费马**伪素数，它**通不过** base-2 **强**（Miller–Rabin）检验，第一步就被拦下；真正能骗过 base-2 强 MR 的最小合数是 $2047$。
- 在并行（候选 2）语境下，强行加一道"全员 base-2 跑完才能开始 Lucas"的**屏障**，反而可能**拖慢**墙钟：它阻止了那个幸运素数所在的线程一口气跑完 base-2 → Lucas，不如"谁先跑完整个 BPSW 谁赢"来得低延迟。
- 这条与第二节末尾、附录 B 已驳回的"BPSW 前加 base-2 预过滤"**同源**：都是把 BPSW 内置的第一步又往外搬一遍。唯一成立的内核——"Lucas 比 base-2 贵，应当短路、base-2 在前"——本就是标准 BPSW 的默认行为，**无需另立为优化**。

**小结：** 把火力对准 BPSW 是对的方向。其中**候选 1（Montgomery）压低单次成本**、**候选 2（多核并行）压低墙钟次数**这两发实打实，合计可让 Regime B 的总时间再降一大截（粗估 $2\!-\!4\times$ 单次 $\times$ 约 $1/\text{核数}$ 墙钟）；候选 3 则是一个换了装的旧冗余。

## 四、复杂度与现实感

- **搜索顺序**：升序，期望只需检查到 $p \approx O((\ln N)^2)$。
- **筛阶段**：预筛 $O(R)$（可跨 $N$ 复用）+ 预计算余数 $O(\pi(R))$ 次大数取模 + 标记一个小区间 $O(B\ln\ln R)$。对千位数全部加起来亚毫秒级。
- **每次重活**：BPSW/MR 约 $O(k\log^2 N)$（乘法用 FFT 可更低），是唯一的大数操作；**用 Montgomery 模约消去除法常数后，单次再降数倍**（见第三·末节候选 1）。
- **总期望开销**：$\approx$ (BPSW 调用次数，约几十) $\times$ (单次 BPSW 成本)，对千位数是**亚秒级**。**全篇真正要优化的就是这个乘积**——左项靠更激进的双筛筛界压低、右项靠 Montgomery 压低。**多核推测式并行**再把墙钟进一步压到 $\approx$ 该乘积 $/\min(\text{核数},\ \text{调用次数})$（降的是延迟，非总功，见候选 2）。
- **诚实的边界**：
  1. 该算法**默认解存在**。哥德巴赫猜想尚未被证明，但已验证到 $4\times10^{18}$ 以上；对一切现实 $N$ 它都能停机出解。它**无限逼近**规律，但**不是猜想的证明**。
  2. 所谓"$O(1)$ 一发命中"不存在：你能把单次扫描压到 $O(1)$，但"要扫多少次 / 要做多少次 BPSW"由素数分布决定，不可消除，只能因为 $p_{\min}$ 极小而显得很快。

## 五、把方案落到机器上：语言与实现选型

前四节是"算法层"。本节回答"用什么写、怎么写不踩坑"，并评判一条流行的 **Rust + rug** 建议。

### 6.1 选型的两条铁律

1. **大数素性测试必须落到 GMP。** 无论用哪门语言，BPSW 这一步都应直接调用 GMP（C 手写汇编 + Montgomery + Toom-Cook，x86 上最快）。Rust 走 `rug`、Python 走 `gmpy2`、C/C++ 直接链 `libgmp`——三者底层是**同一个** GMP，单次 BPSW 的速度**几乎一样**。自己从零手写大数素性测试只在练习时有意义。
2. **并行只为降墙钟延迟，且要绕开 GIL。** 这是第三·末节候选 2 的落地：并行的唯一目的是把"几十次 BPSW"的墙钟压到 $\approx 1/\min(\text{核数},\text{调用数})$。Python 多线程受 GIL 限制需改用**多进程**（`ProcessPoolExecutor`）；Rust/C++ 用共享内存线程即可，无此约束。

> **一个必须澄清的双重计入：** 一旦决定"调用 GMP/rug/gmpy2"，第三·末节**候选 1（Montgomery）就已被白嫖进来了**——GMP 内部本就用 REDC。此时候选 1 **不再是叠加在 rug 之上的额外 2–4×**，而是你已经站在的基线。候选 1 的"质变"只发生在**仍以试除取模的手写路径**上。请勿把它当成独立加速重复计入。

### 6.2 各语言路线的诚实对比（本任务：单个千位 $N$）

| 路线 | 单次 BPSW 速度 | 并行取消的写法 | 环境折腾 | 适合 |
|---|---|---|---|---|
| **Rust + rug** | C 汇编级（=GMP） | `Arc<AtomicBool>` + `rayon`/`crossbeam`，共享内存、无 GIL | 需 GMP/MPFR/MPC 工具链；Linux 容易，Windows 需 MSYS2 | 想要单一静态二进制、无 GC 抖动、最干净的取消语义 |
| **Python + gmpy2** | C 汇编级（=同一个 GMP） | `ProcessPoolExecutor` 绕 GIL；取消靠取消未派发的 future | `pip install gmpy2` 即可 | 最快出原型；本任务亚秒级已足够 |
| **C/C++ + libgmp** | C 汇编级（=GMP） | 手写线程 + 原子标志 | 中等 | 已有 C++ 工程栈 |

**结论：** 在 5600G（6 核 12 线程）/ 16G / 无独显上，**三条路线对单个千位 $N$ 都是亚秒级**。Rust 的真实边际优势是「共享内存取消（免进程开销）+ 无解释器抖动 + 单一二进制」，**不是"快一个数量级"**。所以选型应按"你愿意为哪种工程属性折腾环境"来定，而非迷信某语言是"绝对真理"。

### 6.3 并行骨架（修正版，对照流行伪码的三个坑）

流行建议里的并行伪码有三处需要修正，否则名不副实：

- **坑 1（取消语义被夸大）：** `Arc<AtomicBool>` 只在任务**入口**检查一次，这是**协作式取消（cooperative）**，**不是抢占式（preemptive）**。已经进入 `is_probably_prime` 的线程**不可被中途打断**——GMP 不会轮询你的 flag。真实收益仅为"**让尚未开跑的任务别再开跑**"，对几十个毫秒级 BPSW 足够，但不要宣称"瞬间掐断正在繁重运算的线程"。
- **坑 2（线程数失控）：** `for &p in survivors { spawn(...) }` 是**每候选一线程**，不是 12 线程池；幸存者 50 个就开 50 条线程，与"派发给 12 线程"的注释矛盾。应交给固定大小的线程池（`rayon` 默认按核数）。
- **坑 3（顺序 join 抵消赛跑）：** 按提交顺序 `join` 会先阻塞在 `handles[0]`，与"谁先验出谁赢"的低延迟叙事不符。**想要首完成低延迟就用 `find_any`；想要最小 $p$ 就老实说"升序、命中即取消其余"，二选一。**

修正后的 Rust 骨架（用 `rayon::find_any`，一次表达"首完成即返回 + 自动取消其余 + 线程数=核数"）：

```rust
use rayon::prelude::*;
use rug::{Integer, integer::IsPrime};

/// survivors: 双筛幸存的候选 p（升序，通常仅几十个）
/// 返回任意一个使 N-p 为素的 p；rayon 在某线程命中后会停止派发剩余任务
fn parallel_race(n: &Integer, survivors: &[u64]) -> Option<u64> {
    // find_any: 任一元素满足谓词即返回它, 并尽快停止其余迭代(协作式)
    // 线程池大小默认 = 物理/逻辑核数, 无需手动派发到 12 线程
    survivors.par_iter().find_any(|&&p| {
        let target = Integer::from(n - p);          // N - p, 单次克隆, 相对 BPSW 可忽略
        target.is_probably_prime(25) != IsPrime::No  // BPSW(GMP 已内置) + 额外 MR 轮; reps 取小值即够
    }).copied()
}
```

要点：`find_any` 自带"命中即尽快收手"的协作式取消，省去手写 `AtomicBool`；它**不保证返回最小 $p$**（要最小 $p$ 用 `find_first`，但会牺牲一点并行度）。哥德巴赫只需任意一组解（第三·末节候选 2），故 `find_any` 是对的默认。

### 6.4 与第一节一致的"先串行探一发"优化

第一节的地基是 $p_{\min}$ 极小、**第一个幸存者常常就是答案**（尤其 $N$ 富含小素因子）。因此无脑把整张表 fan-out 是为最坏情况过度配置。更省的做法：**先串行验头 1–2 个（或按核数取一小批），没中再扩大并行**。这把候选 2 的"诚实的边界——推测式并行有轻微超射"落到了实处：常见情形下一发命中、零超射；罕见贫瘠数才付出并行的代价。

```text
function solve(N, survivors):
    # 常见: 富含小素因子, 第一发往往直接命中, 不必开线程
    for p in survivors[:2]:
        if is_prime_BPSW(N - p): return (p, N - p)
    # 罕见: 头几发未中(如 N = 2^k 这类贫瘠数), 才扩大并行
    return parallel_race(N, survivors[2:])   # find_any, 线程数=核数
```

## 六、一句话总结

> 哥德巴赫分拆不是"硬难题"，而是一场**顺序已经选对、只需把每一步做便宜**的搜索。
> 优雅来自认识到：**同余对称性（轮筛）、素因子结构（彗星）、分级测试（漏斗）三者，都是哈代-李特尔伍德密度公式在代码里的不同侧影。**
> 而工程上要先认清自己在哪个 regime：**单个大数的瓶颈永远是那几十次大数素性测试**，不是筛的吞吐——所以把火力对准减少与加速 BPSW，而不是给一个本就免费的筛贴上 SIMD。对准 BPSW 的两发实弹是 **Montgomery 模约**（压单次成本）与**多核推测式并行**（压墙钟延迟）；至于"先 base-2 再统一 Lucas"之类，只是短路 BPSW 的旧冗余、换了身衣服而已。

## 附录 A：实测数据来源

- **最小素数解**：随机 $d$ 位偶数，升序找首个使 $N-p$ 为素的 $p$。10/50/100/300 位各 25 次取中位数与最大值；1000 位单独抽样，最小 $p$ 落在 6133/16223/32191 量级。
- **轮筛存活率**：模 $3\cdot5\cdot7\cdot11=1155$，枚举全部剩余类，统计 $p\not\equiv0$ 且 $N-p\not\equiv0 \pmod{3,5,7,11}$ 的占比。
- **彗星效应**：埃氏筛预处理至 $7\times10^4$，对同量级的 $N$ 统计 $p+q=N$ 的分拆组数。
- 工具：Python + SymPy 1.14（`isprime` 即 BPSW + 强 Lucas），普通容器内运行。
- **关于 GMP 的 BPSW（v3 补注）：** `mpz_probab_prime_p` 自 **GMP 6.2.0** 起内置 Baillie–PSW；**6.1.2 及更早仅做试除 + base-210 Fermat + reps 次 Miller–Rabin，并无 Lucas**。`rug`/`gmpy2` 拿到的是所链接 `libgmp` 的行为，故"白嫖 BPSW"以现代 GMP 为前提。无论哪种，对千位数取 reps≈25 的误判概率都远低于 $4^{-25}$，工程上等价于确定。

## 附录 B：对"极致工程优化"建议的逐条评判

| 建议 | 评判 | 理由 |
|---|---|---|
| 动态筛界 $r_{\max}=B/20$ | ⚠️ **方向反了** | Regime B 里 BPSW 极贵、标记极廉，应**激进**筛（$R\sim10^4{-}10^6$）；且 $R$ 与 $B$ 本是独立量，不应绑定 |
| 轮筛替代逐数遍历 | ➖ 无害的小常数优化 | 对 $p$ 的筛已包含此效果；瓶颈不在此，价值有限 |
| Bitset 取代 bytearray | ➖ 小优化 | 小区间下两者都全程驻留缓存，bytearray 更简单 |
| 区间撑到 262144 对齐 L1（号称 3–5×） | ❌ **用错且有害** | 提前终止使解几乎总在区间最前端；撑大区间是过度计算，"3–5×"是 Regime A 结论 |
| SIMD 位与一次算满块幸存者 | ❌ **用错瓶颈** | 提前终止要低延迟而非高吞吐；在小块上不必要，在大块上是优化被浪费的工作 |
| 预计算 $N \bmod r$、筛内全 64 位化 | ✅ **正确，已采纳** | 大数取模是筛阶段唯一大数操作，一次算好复用最优 |
| BPSW 前加 base-2 强伪素数预过滤 | ❌ **冗余** | BPSW 第一步本就是 base-2 强 Fermat，前置等于做两遍，零收益 |
| **蒙哥马利模约（Montgomery / REDC）** | ✅ **正确，已采纳** | 模数 $N-p$ 单次测试内固定且为奇，前提恰好成立；除法→移位+乘+按位与，单次 BPSW 降约 $2\!-\!4\times$。注：GMP/gmpy2 已内置，质变主要在手写 C++/Rust 路径 |
| **多核推测式并行"彩票扫荡"** | ✅ **正确，已采纳** | 哥德巴赫只要任意一组解、不要求 $p$ 最小，故可并行验队首幸存者、首中即取消其余；墙钟降至 $\approx 1/\min(\text{核数},\text{调用数})$。诚实标注：降延迟非降总功，略有推测超射 |
| **非对称 BPSW：先全员 base-2、再统一 Lucas** | ❌ **基本冗余** | 正确的短路 BPSW 本就只在 base-2 通过后才跑 Lucas，随机大合数已被 base-2 强 MR 当场拦下，重排省不下 Lucas；且 $341$ 失于 base-2 强 MR（最小 base-2 强伪素数是 $2047$），并行下加 base-2 屏障反而可能拖慢——与"base-2 预过滤"同源 |
| **选 Rust + rug（或 gmpy2 / libgmp）跑 BPSW** | ✅ **正确，应采纳** | 三者底层同一个 GMP，单次 BPSW 同为 C 汇编级；现代 GMP（≥6.2.0）的 `is_probably_prime` 即 BPSW（≤6.1.2 仅 MR）。注：选它即已白嫖候选 1（Montgomery），勿再叠加计入 |
| `Arc<AtomicBool>` 入口检查"瞬间掐断繁重线程" | ⚠️ **措辞错误** | 这是**协作式**取消（任务边界检查一次），**非抢占式**；已进入 GMP 调用的线程不可中途打断。真实收益=让未开跑的任务别再开跑。用 `rayon::find_any` 更直接 |
| 并行伪码"每候选一线程"+ 顺序 join | ⚠️ **名不副实** | 每候选一 `spawn` 不是 12 线程池；顺序 `join` 阻塞在首个 handle，抵消"首完成即赢"。应用固定线程池 + `find_any`（任意解）或 `find_first`（最小 $p$） |
| 无脑把整张幸存者表 fan-out 并行 | ➖ **过度配置** | 由第一节 $p_{\min}$ 极小，第一发常直接命中；应先串行探 1–2 个、未中再并行，以消除常见情形下的推测超射 |
